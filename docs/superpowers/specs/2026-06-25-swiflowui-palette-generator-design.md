# SwiflowUI Palette Generator Design

> **Date:** 2026-06-25 ┬Ę **Status:** approved, ready for implementation plan
> **Milestone:** SwiflowUI **M8 (1.1) ŌĆö Token correctness & generation** (second of three
> bullets; the contrast tokens shipped in PR #66).
> **Origin:** the [Reshaped evaluation](../../future-work/swiflowui-reshaped-evaluation.md)
> steal-list #2 (OKLCH palette generator) + #3 (OKLCH color space).

## Problem

Apps re-skin SwiflowUI by hand-overriding `--sw-*` tokens (a `<style>` block in `<head>`; see
`docs/guides/swiflowui-theming.md`). For a brand color, that means hand-picking not just
`--sw-accent` but a coherent, **contrast-safe** family across light *and* dark: the dark-mode
accent, hover/active steps, the readable on-accent text, and the on-tint "strong" text. Getting
all of that right by eye is exactly the hand-guessing M8 set out to remove ŌĆö and nothing warns
an app when its chosen brand color produces a sub-AA result.

## Goal

Let an app theme by **seed color**: `swiflow theme --primary "#7c3aed"` emits a small,
correct `--sw-*` override the app includes, and **validates** that the resulting accent family
meets WCAG (failing the build if not). Two enabling changes make the override tiny and the
result automatically coherent:

1. **The accent family derives from `--sw-accent`** (this design's P1), so re-pointing one token
   cascades hover/active/text/strong.
2. **A build-time CLI** (P2) derives the dark-mode accent from the light seed and validates the
   whole family ŌĆö reusing the M8 color pipeline, promoted into a shared native target.

## Non-goals (deferred)

- **Neutrals / full palette** (`--sw-bg`/`--sw-surface`/`--sw-text`/`--sw-border`): a later,
  larger milestone (deriving contrast-proven neutrals, possibly brand-tinted).
- **`--danger`/`--success` seeds:** semantics rarely rebrand; accent-only is the 80% case.
- **p3 upgrade for the generated accent:** the default theme keeps its hardcoded `display-p3`
  block; an app override shows the seed in sRGB on p3 displays (acceptable). Add later.
- **Runtime / dynamic (user-selectable) theming:** the generator is build-time; no color math
  ships in the wasm (preserves the M8 decision).
- **Multiple themes / theme files in one run.**

## Architecture

Two product changes plus a module restructure.

| Piece | What | Where |
|-------|------|-------|
| **P1** | `--sw-accent-hover`/`-active` derive from `var(--sw-accent)` | `Sources/SwiflowUI/Theme.swift` (ships) |
| **P2** | `swiflow theme --primary <hex>` command | `Sources/SwiflowCLI/` |
| **Shared** | Promote the M8 color pipeline out of the test target | new `Sources/SwiflowColor/` |

**`SwiflowColor` is native-only.** `Sources/SwiflowCLI` (the build tool) and
`Tests/SwiflowUITests` (the M8 proofs) depend on it; the wasm `SwiflowUI` library does **not**.
So the OKLab/contrast math never enters a shipped wasm bundle ŌĆö the same constraint M8 honored.

## P1 ŌĆö accent family derives from `--sw-accent`

In `baseStyleSheet`, replace the literal `--sw-accent-hover`/`-active` declarations with
derivations (the per-mode lightness flip is expressed by pairing a darken arm with light mode
and a lighten arm with dark mode inside `light-dark()`):

```css
--sw-accent-hover:  light-dark(oklch(from var(--sw-accent) calc(l - 0.08) c h),
                               oklch(from var(--sw-accent) calc(l + 0.08) c h));
--sw-accent-active: light-dark(oklch(from var(--sw-accent) calc(l - 0.16) c h),
                               oklch(from var(--sw-accent) calc(l + 0.16) c h));
```

`var(--sw-accent)` resolves to the current scheme's value, so the light arm darkens the
light accent and the dark arm lightens the dark accent ŌĆö matching today's intent (hover/active
darker in light, lighter in dark). A progressive-enhancement literal fallback is kept first
(same pattern as M8's `-strong`), so pre-`oklch(from)` browsers keep working values.

`--sw-focus-ring` is already `var(--sw-accent)` ŌĆö unchanged. `-text` and `-strong` already
derive from `var(--sw-accent)` (M8) ŌĆö unchanged.

**Net effect:** overriding `--sw-accent` alone re-skins the entire accent family. **Visible
caveat:** the *default* theme's hover/active recompute from their hand-tuned hexes to the
`calc`-stepped values ŌĆö a subtle shift; hover/active are non-text so there is no contrast risk;
verified by eye in the demo. The step constants (0.08 / 0.16) are tunable during implementation.

## P2 ŌĆö the `swiflow theme` command

**Invocation:** `swiflow theme --primary "#7c3aed" [--out path]`
- `--primary <hex>` (required): the light-mode accent seed (`#rgb` or `#rrggbb`).
- `--out <path>` (optional): write to a file; default is stdout (`> theme.css`).

**Output:** a commented CSS block ŌĆö the same shape apps write by hand today:

```css
/* Generated by `swiflow theme --primary #7c3aed`. Include after SwiflowUI's styles.
   Re-points --sw-accent; hover/active/text/strong derive from it automatically. */
:root {
  --sw-accent: light-dark(#7c3aed, #a78bfa);
}
```

(The dark arm `#a78bfa` is derived ŌĆö see Algorithm.) The app includes this where manual token
overrides go (a `<style>`/`<link>` in `<head>`); custom-property resolution means
`-hover/-active/-text/-strong` read the cascaded `--sw-accent`, so the whole family follows.

**Validation (the real value-add):** before emitting, the command recomputes the derived family
for the seed in both modes using `SwiflowColor` and checks WCAG:
- **`--sw-accent` used as text/links** (ghost buttons, links) on the surface Ōēź **3:1** (the
  UI/large-text bar). This is the check that actually catches a washed-out brand color ŌĆö e.g. a
  light yellow is ~1.07:1 as text on white. The 3:1 (not 4.5) bar is deliberate: it allows the
  conventional blue-on-white link/ghost pattern (the default `#3b82f6` is 3.68:1) while rejecting
  genuinely unreadable accents.
- `-text` `contrast-color()` result Ōēź 4.5 on the accent fill (the Baseline path).
- `-strong` (the `oklch(from)` derivation at the shipped lightnesses) Ōēź 4.5 on the 15% tint
  (and Ōēź 7 under `prefers-contrast: more`).

The latter two are **robust by construction** ŌĆö `contrast-color()` mathematically can't fall
below ~4.58:1, and the `oklch(from)` derivation pins lightness ŌĆö so they rarely trip; they're a
regression guard. The accent-as-text check carries the real coverage.

**Legacy-fallback gap (deliberate):** validation does **not** gate on the static
`--sw-accent-text` fallback (`#0b1220`). That fallback is tuned for the default light accent and
is sub-AA on *medium-dark* custom accents (violet `#7c3aed`, indigo) ŌĆö but `contrast-color()`
(Baseline 2026) correctly flips those to white, so the rendered result is AA on every modern
browser. Gating on the pre-Baseline-only fallback would reject common brand colors for an
audience that barely exists, so we don't. The trade-off: a violet-themed app viewed on a
pre-`contrast-color` browser gets a sub-AA primary-button label. Accepted; documented here. On any failure the command
prints a specific diagnostic (token, mode, measured ratio, target) and **exits nonzero**, so a
poor brand color fails a build rather than silently shipping sub-AA. On success it exits zero and
emits the CSS.

## Dark-accent derivation algorithm

From `OKLCH(seed)` = (L, C, H):
- `darkL = clamp(L + 0.10, lower: 0.68, upper: 0.76)`
- `darkC = C * 0.78`
- `H` unchanged
ŌåÆ convert to sRGB, gamut-clamp, format as `#rrggbb`.

This roughly reproduces the existing `#3b82f6 ŌåÆ #60a5fa` pairing (lighter, slightly less
saturated dark-mode accent). The constants are tunable; the validation pass is the safety net,
so a tuning change that breaks contrast fails loudly.

## Module restructure

- **Create `Sources/SwiflowColor/`** (a native-only internal library): move the M8 pipeline
  (`ContrastColor.swift` ŌĆö `LinRGB`, `OKLab`, `OKLCH`, `Color` with hex/luminance/WCAG/
  `mixOKLab`/`oklchFrom`/`contrastColor`/conversions) here; add `darkAccent(from:)` and the
  family-validation helpers.
- **`Package.swift`:** add the `SwiflowColor` target; make `SwiflowCLI` and the `SwiflowUITests`
  test target depend on it. `SwiflowUI` (wasm) gains **no** dependency on it.
- **`Tests/SwiflowUITests`:** delete the local `Support/ContrastColor.swift`; import
  `SwiflowColor`. The M8 contrast proofs (`ThemeContrastTests`) keep passing unchanged.
  `Support/CSSValueParsing.swift` stays test-only (it parses the emitted sheet ŌĆö not needed by
  the CLI, which validates from the seed directly).

## Testing

- **`SwiflowColor` unit tests:** `darkAccent(from:)` is deterministic and roughly reproduces the
  default pair for `#3b82f6`; the family-validation helper returns "pass" for `#3b82f6` and
  "fail (which token, ratio)" for a known-bad seed (e.g. a light yellow `#facc15`).
- **CLI test:** `swiflow theme --primary "#3b82f6"` emits the expected `:root` block and exits
  zero; `--primary "#facc15"` prints a validation diagnostic and exits nonzero; missing
  `--primary` errors with usage.
- **`ThemeTests`:** assert `--sw-accent-hover`/`-active` now contain `oklch(from var(--sw-accent)`
  with the `calc(l` step (and keep their literal fallback); existing assertions stay green.
- **Demo eyeball:** the default hover/active shift is acceptable; a themed include re-skins
  buttons/links coherently. (CI skips example builds ŌĆö build the demo locally.)
- `swift test` green; `swiflow theme` golden output stable.

## Decisions resolved during brainstorming

1. **Generator model** ŌåÆ **build-time CLI** (`swiflow theme`), not a runtime API ŌĆö keeps color
   math out of the wasm.
2. **Scope** ŌåÆ **accent family only**; neutrals/semantics/full-palette deferred.
3. **Hover/active** ŌåÆ **make them derive from `--sw-accent`** so re-pointing one token cascades
   the whole family (and the generator emits just `--sw-accent`); accepted the subtle shift in
   the default theme's hover/active.
